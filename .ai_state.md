## 🧠 Dev Notes & Hypotheses

- 10-07-2026 (Sonnet 5, `claude-sonnet-5`, H498): `write-translations --locale en` was used to scaffold the UI-chrome JSON skeletons (navbar/footer/code.json/sidebar-category current.json) rather than hand-guessing Docusaurus's translation-key naming convention — safer and matches what the framework will actually look up.
- Internal `/atlas/` links (in both `atlas-docs/intro.md` and its EN counterpart) were switched from root-relative (`/atlas/theme/slug`) to file-relative-with-extension (`./theme/slug.mdx`) — the former resolves relative to the site root regardless of active locale (breaking under `/en/`), the latter resolves relative to the source file and is locale-correct. This generalizes the fix H486/H490 already applied ad hoc to a couple of links; worth remembering for any future hand-authored cross-link in `atlas-docs/` or `data-docs/`.
- `docusaurus.config.mjs`'s global `title`/`tagline` are evaluated once and are NOT automatically per-locale — each locale's full production build (`[INFO] [en] Creating an optimized production build...`) reuses the same static config object. Per-page titles ARE correctly localized (they come from each MDX file's own frontmatter/`<Translate>` calls); only the trailing site-name suffix Docusaurus appends to `<title>` stays Russian on EN pages. Not fixed this pass (cosmetic, out of H498's explicit scope) — flagged as a residual in Next Steps.
- Mint-collision note: `mint_handoff.py`'s claim-push for H498 reported "could not push... only locally claimed" but the content landed on `origin/main` anyway (confirmed via a later `git ls-tree origin/main`) — the warning appears to be conservative/racy under heavy concurrent-session load on Uprava that day; don't assume a WARN means the push definitely failed, verify against origin before re-minting.
- (H486/H490 dev notes from prior sessions trimmed — see git history of this file for the full record if needed; the ones still load-bearing were folded into README.md/CHANGELOG.md at the time.)
- **12-07-2026 (Opus 4.8, `claude-opus-4-8`, H832) — H486 silently corrupted an atlas HTML file; audit the others.** `afanasy_concordance_index.html` was born broken in the H486 Docusaurus rebuild (`f54d64d`): a truncated theme-toggle `<button style="…box-sh">` swallowed its script includes + the whole inline `<script>`, and a duplicated 123-line block was spliced past the IIFE close (`})();m: 8px;">`). `document.scripts` was empty → the page rendered no list/search/detail, standalone *and* iframed — undetected for two days. A quick sweep found this was the **only** page with the `box-sh…(function` splice, but the failure mode (a rebuild/transform silently mangling a static widget so it dies at runtime with no build error) is worth a periodic `node --check` sweep of the inline scripts across `static/atlas/*.html`, since the `data-validate`/`validate_widgets` CI gate only checks data-bundle drift, not that each widget's inline JS actually parses. Detection recipe: `for f in static/atlas/*.html; do extract inline <script>, node --check</script>; done`.
